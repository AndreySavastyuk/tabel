# Табель СКУД — контекст и инструкции для Claude Code

Веб-табель учёта рабочего времени по данным СКУД (системы контроля доступа).
Стек: **FastAPI** (`api/`) + чистый движок расчёта (`engine/`) + **React/Vite** (`web/`),
миграции **Alembic** (`alembic/`). Регрессионный оракул легаси — `legacy/` (заморожен, не трогать).
Дорожная карта и статус фич — `ROADMAP.md`.

## Рабочая ветка
Основная (и единственная) линия работы — **`main`**. Веди разработку на `main`; она
синхронизируется между домашним и рабочим компьютером через GitHub
(`https://github.com/AndreySavastyuk/tabel`).

## Синхронизация между машинами (дом ⇄ работа)

**Когда пользователь говорит «актуализируй проект» / «подтяни проект»:**
1. `git fetch origin`, затем `git pull --ff-only origin main` (если есть локальные коммиты —
   `git pull --rebase origin main`).
2. Если изменились зависимости (`requirements.txt` или `web/package-lock.json`) — запустить
   `setup.bat` (или вручную: `.venv\Scripts\python -m pip install -r requirements.txt`; в `web/` — `npm ci`).
3. Применить миграции БД: `.venv\Scripts\python -m alembic upgrade head`.
4. Прогнать быстрые проверки (см. ниже) и кратко отчитаться: что подтянулось, какие миграции применены.

**Когда пользователь говорит «сохрани / запушь работу»:**
1. Прогнать проверки качества (см. ниже) — они должны быть зелёными.
2. Стейджить БЕЗОПАСНО: `git add -A` затем **`git reset -- data`**; убедиться, что
   `git diff --cached --name-only` НЕ содержит `data/`, `*.db`, `.env`, выгрузок СКУД.
3. Коммит с осмысленным сообщением + `git push origin main`.

## Персональные данные — НЕ в git (критично)
В репозитории НЕТ ПДн. `.gitignore` исключает: `*.csv`, `*.xlsx`, `*.txt` (кроме `requirements.txt`),
`ЛЭЗ/`, `*.db`, `.env`, `_uploads/`, `data/`.
- **НИКОГДА не выполнять `git add data/`** — там реальные выгрузки СКУД (ФИО/проходы) и посторонние `.py`.
- `tabel.db` (рабочая БД: пользователи + обработанные данные) НЕ синхронизируется через git.
  Перенос реальных данных между машинами — скопировать `tabel.db` вручную (USB / личное облако)
  в корень проекта.
- Альтернатива на свежей машине без БД: положить исходные файлы СКУД в корень
  (`StorK.csv`, `SIGUR.xlsx`, `ЛЭЗ/lez.xlsx`, `ЛЭЗ/Сотрудники.txt`, справочники) и выполнить
  `.venv\Scripts\python -m scripts.seed_from_excel` — создаст справочники, календарь и дев-пользователей.

## Первая настройка машины и запуск
- **Первый раз на машине:** `setup.bat` (создаёт `.venv`, ставит зависимости Python и web, применяет миграции).
- **Запуск приложения:** `run.bat` — поднимает API на `:8000` и Web на `:5173`, открывает браузер.
- **Дев-логины (только dev):** `admin/admin` (Кадры/Админ), `buh/buh` (Бухгалтер), `ruk/ruk` (Руководитель отдела).
- API-доки в dev: `http://127.0.0.1:8000/docs`. Фронт проксирует `/api` → `:8000`.

## Проверки качества (прогонять перед коммитом)
- Бэкенд-тесты: `.venv\Scripts\python -m pytest -q` (ожидаемо ~94 passed, 1 skipped; 1 skip —
  отсутствие данных по отделам, оставлено намеренно).
- Python-линт: `.venv\Scripts\python -m ruff check api engine scripts tests alembic` (clean).
  НЕ запускать `ruff check .` — в корне легаси-дубли, которые не отключаются по имени.
- Фронт-линт: в `web/` → `npm run lint` (oxlint `--deny-warnings`, ноль предупреждений).
- Фронт-сборка/типы: в `web/` → `npm run build` (`tsc -b && vite build`).

## Соглашения проекта
- Движок `engine/*` НЕ менять разрушительно — есть golden/parity-тесты против `legacy/` (байт-в-байт).
  Логика «над движком» (период, отклонения, объяснение, закрытие) живёт в `api/`-слое.
- Миграции: только через `op.batch_alter_table` (SQLite-совместимость), строковые «энумы» `String(12)`,
  Python `default=` (не `server_default`). Текущий head — см. `alembic heads`.
- Vite 8 (rolldown) держит `.node` бинарь занятым во время `npm run dev` — остановить dev-сервер
  перед `npm ci`/переустановкой, иначе EPERM.
